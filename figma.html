Claude + Figma — Best Practices

A practical guide for designers, PMs, and developers using Claude alongside Figma.

The core idea

Figma AI works inside the canvas — it generates layouts, rewrites copy, removes backgrounds. Claude works around the canvas — it thinks, writes, explains, and prepares. The two complement each other; they don't overlap much.

Figma AI in-canvas tasks: copy variants, auto-layout, visual cleanup
Claude thinking, writing, explaining, and anything that requires context

For developers, there's a dedicated Figma plugin for Claude that connects the two tools directly — Claude can read your Figma files, extract components, tokens, and layout data, and translate designs into production code.

For developers: the Figma plugin for Claude

Install the plugin at claude.com/plugins/figma. It gives Claude direct access to your Figma files so you don't have to copy-paste anything manually.

What it can do

  • Extract structured design data (layout, typography, colors, spacing) from any Figma frame
  • Read design variables and tokens for consistency with your codebase
  • Map Figma components to existing code via Code Connect
  • Capture visual references for design validation

Key commands

  • /implement-design — paste a Figma URL, get production-ready code using your design system
  • /create-design-system-rules — generates custom guidelines encoding your team's conventions
  • /code-connect-components — maps Figma components to their codebase counterparts

Example prompts

  • "Implement this Figma design: [paste Figma URL]"
  • "Extract the color variables from this design file"
  • "Connect my Button component to its Figma counterpart"
  • "Create design system rules for my React project"

The full loop this enables: Figma design → Claude reads it → code generated → developer adjusts → design validated against implementation. No more manual handoff or screenshot-based reviews.

Going further — bidirectional sync (advanced)

The traditional one-way handoff (Figma → code) is increasingly obsolete. Using Figma's MCP, teams can extract live interfaces from production, staging, or localhost back into Figma, collaborate on changes, then push back to code. This prevents design drift — designers always work from actual implemented state, not outdated specs.

Source: Gui Seiz & Alex Kern, Figma — "From Figma to Claude Code and back", Lenny's Newsletter

Advanced workflows (developer + designer together)

These workflows come from how Figma's own team operates. They require Claude Code + Figma MCP set up, but represent where the industry is heading. (Source: "How Figma team syncs design and code with Claude Code and Codex", ChatPRD)

Workflow 1: The code-to-design-to-code sandwich

⚠ Problem: Design and production code drift apart. Designers end up redesigning things that are already built differently.

How it works

  1. Run your local app (or point to staging/production)
  2. Open Codex connected to Figma's MCP
  3. Prompt: "send the [page name] to Figma"
  4. Production code converts to fully editable Figma layers (vectors, text, shapes)
  5. Designer makes changes directly in Figma — colors, layout, copy, spacing
  6. Copy the updated Figma frame URL
  7. In Claude Code, prompt: "bring the changes from this Figma frame into my codebase — [paste URL]"
  8. Claude analyzes the diff and generates code patches. App updates to match Figma exactly.

✓ Designers work on actual implemented UI, not a guess. Developers get precise diffs, not vague feedback. No manual CSS translation.

Workflow 2: Batch export all component states

⚠ Problem: Designers work blind — they only see the happy-path state, not error, loading, empty, or edge-case states.

How it works

  • Engineer runs a single Claude Code command: "send all [N] states of the [component] to Figma"
  • Claude inspects the codebase, identifies all component variations (default, error, success, loading, disabled, etc.)
  • Exports all states as individual editable components in Figma, arranged side-by-side

✓ Designers can refine what users actually experience, not just the ideal flow. Catches edge-case UX issues before they ship.

Workflow 3: Automate your pre-ship checklist with a /ship skill

⚠ Problem: Pre-merge checklists are repetitive and error-prone when done manually.

How it works

  • Create a Claude skill called /ship that encodes your team's standard pre-merge process
  • When triggered, it automatically: runs lint, runs build, pushes to Git, monitors CI/CD, auto-fixes minor lint errors (up to 5 retries), and surfaces anything that needs human attention
  • Safety guardrails built in: no force pushes, no credential commits

✓ Engineer types /ship and walks away. The same principle applies to any repeatable workflow — design review checklists, handoff steps, accessibility audits.

Before you open Figma

This is where Claude saves the most time.

Turn a vague brief into a structured one

Paste the raw requirement or client email. Ask Claude to extract: the user goal, the constraints, the edge cases to design for, and the open questions you need to answer before starting.

Draft all copy before designing

Writing copy inside Figma is slow. Write it in Claude first. Give Claude: the screen name, the user's goal at that moment, the desired tone. Ask for 2–3 options. Pick one, then design around it.

Map user flows in words

Describe a feature. Ask Claude: "What are all the states this screen can be in?" or "What can go wrong in this flow?" You'll catch edge cases before you've drawn a single frame.

Research quickly

Ask Claude how similar products handle a UX pattern, what accessibility requirements apply, or what standard microcopy looks like for a given action (e.g. form errors, empty states, confirmation dialogs).

While designing

Generate realistic placeholder content

Fake Latin text makes designs look fake. Ask Claude for realistic names, addresses, product descriptions, error messages, or notification copy that matches your actual use case.

Write and refine UI copy

Paste your draft copy. Tell Claude: the character limit, the user's context, the tone. Ask it to rewrite, shorten, or generate alternatives. Iterate in Claude — paste the final version into Figma.

Name components and design tokens

Paste your component list and ask Claude to suggest a consistent naming convention. Or paste your color/spacing values and ask for naming that fits a design system.

Check accessibility

Describe a screen in words (layout, colors, interactive elements). Ask Claude: "What accessibility issues might this have?" It won't replace a proper audit but will catch obvious problems early.

Write interaction notes

Describe a micro-interaction. Ask Claude to write the developer handoff note for it — trigger, behavior, animation, edge cases. Paste directly into your Figma annotation.

After designing

Write the handoff document

Paste your component/screen list. Ask Claude to generate a structured handoff note: what each component does, its states, how it responds to interactions, and open questions for the dev team.

Prepare client presentation narrative

Paste your design decisions. Ask Claude to write a short narrative explaining the rationale — "we chose X because the user at this point is Y." Saves 30–60 minutes of prep before every presentation.

Respond to feedback

Paste the client or stakeholder feedback. Ask Claude to help you draft a structured, professional response: what you'll change, what you won't and why, and what questions you need answered before proceeding.

Write a design decisions log

After each major design sprint, paste your key decisions into Claude and ask it to format them as a decisions log. Over time this becomes invaluable context for onboarding, audits, and client conversations.

Prompting tips

Give context, not just the task

Bad"Write button copy."
Good"Write button copy for a confirmation screen where the user is about to delete their account. Tone: serious but not alarming. Max 3 words."

Tell Claude what you already have

Paste your existing draft. "Here's my current version — make it shorter and more direct." Claude does better when it has something to react to.

Ask for options

"Give me 3 versions" takes the same time and gives you something to choose from, compare, or combine.

Set constraints up front

Character limits, tone, platform (mobile vs desktop), user type. Put these at the top of your prompt, not the end.

Iterate, don't restart

Keep the conversation going. "Make it warmer." "Now shorten it by half." "Add a sense of urgency." Claude holds the context — you don't need to re-explain each time.

Know when NOT to prompt

For precision work — exact hex codes, specific spacing values, pixel-perfect adjustments — use Figma's visual tools directly. As Gui Seiz (Figma designer) puts it:

"No one wants to prompt for the exact hex code or shade of yellow."

AI is for thinking and generating; direct manipulation is for precision.

Automate your repeatable workflows

If you have a standard process you run repeatedly (pre-launch checks, handoff steps, component review), turn it into a Claude skill. Figma's own engineers built a /ship skill that automates Git pushes, CI monitoring, and lint fixes. The same principle applies to design workflows.

Structure your codebase for AI readability (for developers)

Figma's engineers found that spending 20–30% of development time restructuring code for AI legibility significantly improves Claude's output quality. Clean, well-named, modular code gets better AI-generated results — not because Claude needs perfect code, but because it can understand context and patterns more reliably.

Source: ChatPRD — "How Figma team syncs design and code"

What NOT to put into Claude

Follow the paste test: if you wouldn't post it in a public Slack channel, don't put it in Claude unless you're on an enterprise plan with a signed DPA.

Never put in

  • Client personal data (user names, emails, IDs)
  • Confidential business data (revenue figures, unreleased product plans)
  • API keys, credentials, or access tokens

Safe to put in

  • Design copy and UI text
  • Feature descriptions and user flows
  • Generic design questions
  • Your own drafts and notes
  • Publicly available information about the product
